## 1. 【最優先・要中止】既存キャッチコピー「学校より学べて公園より楽しくて会社より稼げる」

**すでに本番環境で稼働中**。`frontend/public/index.html` の6箇所（L3521, L7341, L7352, L9740, L10157, L12944）のチュートリアル文言・FAQ回答・マスコット台詞。加えて03の成果物B・Eの全テンプレート（7日分すべて）で再利用が計画されていた。

### 問題点

- 「会社より稼げる」は、SchoolPark/Emuでの活動が**特定可能な比較対象（会社勤務の給与）より経済的に有利**であると断定する表現であり、景品表示法5条2号（有利誤認表示）に極めて抵触しやすい典型パターン。実態としてEMUERは ①現時点で暗号資産・有価証券ではない ②外部換金・他資産との交換ができない ③価値を運営が保証しない（`terms.html` 第5条）という設計であり、「会社の給料より稼げる」という経済的優位性を裏付ける根拠が存在しない。
- 消費者庁は「副業」「稼げる」を訴求する広告について、実態を伴わない収益期待表示を継続的に問題視している（特商法・景表法双方で摘発例あり）。ユーザー層に未成年が含まれうる構造（`terms.html` は18歳以上または法定代理人同意を要件とするが、実際のマスコット文言はチュートリアル全ユーザーに表示）である点も、行政・消費者からの視線が厳しくなる要素。
- **資金決済法との関係**: EMUERを「稼げるお金」として訴求すればするほど、社内的にEMUERを「暗号資産ではない」と整理している `terms.html` 第5条の前提（限定的な自社内ユーティリティであり資産性を持たせない）と、対外的なマーケティング表現が矛盾する。将来的に規制当局から見解を求められた場合、「実質的に金銭的価値を持つものとして販売・訴求していたのではないか」という反論の口実を自ら作ることになり、資金決済法上の立場を弱める。**景表法だけでなく、EMUERの法的性質そのものを守るために使うべきではない。**

### 修正文言案

- 主案: `学校より学べて、公園より楽しくて、会社より夢中になれる場所。`
- 代替①: `学校より学べて、公園より楽しくて、毎日が自分の物語になる場所。`
- 代替②（短縮向き）: `学校より学べて、公園より楽しめる場所。`

## 2. タグライン#6「読めば知る、書けば報われる。」— 要修正

Emuは投稿するとEMUERが付与されうる仕組みであるという前提のもとで「書けば報われる」と表示すると、一般消費者は「投稿＝経済的対価」という因果関係を想起する。

しかし `backend/server.js` 確認の結果、EMUERの付与は「投稿すれば自動的にもらえる」設計ではない:

1. 無料パス取得時の初期付与（1,000 EMUER、投稿とは無関係）
2. `emuer_rate` 登録済み全ウォレットへの月次一律配布（投稿の有無を問わない）
3. 運営裁量による個別ボーナス送金（`admin/send-reward`）

「執筆」と「報酬」の間に自動的・確実な連動がなく、景表法5条2号の**有利誤認表示**に該当するリスクが高い。「報われる」を精神的・社会的な意味で使うつもりでも、EMUERの存在を知りうるユーザーには経済的意味に読める点が問題の核心。

**修正案**: `読めば知る、書けば届く。`（推奨）／`読めば知る、書けば、つながる。`／`読めば知る、書けば、誰かに響く。`

## 4. AIキュレーターの表示・開示

### 4-1. 表示3層の過不足 — 概ね問題なし。ただし実装の完全性がリリース条件

設計（プロフィール文＋バッジ＋機械的開示フッター）は妥当。特に「LLMの出力に開示文を書かせず、コード側で機械的に必ず連結する」設計は、ハルシネーションで開示文言自体が欠落・改変されるリスクを排除しており評価できる。

ただし02自身が指摘している落とし穴（`getUserTypeBadge()` が `ai_curator` を認識しない限り、AI投稿がデフォルトの「ゲスト利用者」バッジで表示される）は、法務的には単なる実装バグではなく**「正体を隠した表示」を生み出す景表法・消費者保護上のクリティカルな欠陥**。

**判定: 要修正（実装完了をAIキュレーター稼働のリリース条件とする）**
- `getUserTypeBadge()`（`index.html` 11113行）へのcase追加を、本番投稿開始**より前**にデプロイ・検証すること。
- 「AI投稿を1件手動生成し、フロントで実際にAIバッジが表示されることを目視確認してから自動運転を開始する」手順を運用マニュアルに明記。

### 4-2. 景表法・ステルスマーケティング規制（2023年10月施行告示）— 問題なし

ステマ規制（景表法5条3号に基づく指定告示）が禁じるのは「事業者の表示であるにもかかわらず、一般消費者がそれを判別することが困難な表示」。本件はプロフィール・バッジ・本文末尾の三重の自己申告構造であり、「第三者の自発的な意見を装う」という定義要件（事業者の関与の秘匿）にそもそも当たらない。**「公式」「AI」であることを隠さず名乗っている自社発信はステマ規制の対象外。**

**どこから危険になるかの線引き**:
1. AI司書に「一人称の体験談」「個人の感想」を語らせた瞬間に、人間オーナーの投稿枠と混同され事実上のステマに近づく。→ 02のNGルール「個人の体験談を語らない」は**法務的に必須要件**として維持すること。
2. 将来的に**匿名・非公式風の追加AIアカウント**（一般ユーザーを装うAI）を作る場合は、今回の設計とは全く別の高リスク領域。**現行設計を拡張する形での「非公式風AIアカウント」は今後も作らない**ことを社内ルール化すべき。
3. AI司書がEmu自体やEMUERを「おすすめ」する投稿を始めた場合、公式アカウントの自画自賛広告としてステマとは別枠で優良誤認規制の対象になる。現行の担当領域切り分けを維持する限りこの懸念は生じない。

### 4-3. 弁護士法72条（法律事務）— 要修正（免責文言の強化）

特定の依頼者の個別紛争・個別事案に対する法的判断の提供（鑑定・代理・仲裁・和解の斡旋等）を「業として」有償で行うことが72条の規制対象。不特定多数に向けた**公開情報の解説・歴史的背景の紹介**（新聞・書籍・ニュース番組の法律コーナーと同種）は原則として「法律事務」に当たらない。設計のガードは方向性として正しい。

ただし現行の開示フッター文言は簡潔すぎるため、以下に差し替え（**02に反映済み**）:

```
※本投稿は公開情報の教養としての紹介であり、法律相談・税務相談・投資助言・投資勧誘には該当しません。個別の法的判断や資産運用の意思決定は、弁護士・税理士・ファイナンシャルプランナー等の専門家にご相談のうえ行ってください。内容の正確性には努めていますが、最新の法改正等を反映していない場合があります。誤りに気づかれた場合は運営お問い合わせ窓口までご連絡ください。
```

コード側固定文のため、生成コスト（トークン）には加算されない。

### 4-4. 金融商品取引法（投資助言業）— 概ね問題なし

「投資助言業（金商法2条8項11号）」は、特定の投資判断について契約に基づき対価を得て個別に助言することが要件。不特定多数への無償の一般的制度紹介（NISA・iDeCoの仕組み説明等、個別銘柄・商品を挙げない）は書籍・マネー誌の解説記事と同種であり原則該当しない。

**追加ガード（システムプロンプトに追記、02に反映済み）**:
```
5. 利回り・リターンの数値を予測・保証する表現をしない（「年利〇%が期待できる」等）。
6. 「今が始め時」「絶対に得をする」等、断定的な時機・優劣の判断を示す表現をしない。
7. 特定の金融機関名・商品名・ファンド名を挙げない。
```

### 4-5. 著作権 — 要修正（軽微・運用ルールの明記強化）

| 対象 | 評価 |
|---|---|
| ①青空文庫（著作権切れ） | 問題なし。ただし `ai_curator_sources` シード段階で「PD確定済みの青空文庫収録作品のみを対象とする」登録基準をドキュメント化すること |
| ②法令原文の引用 | 問題なし。著作権法13条1号により法令に著作権は生じず自由利用可 |
| ③官公庁サイトの解説文 | 設計通り「事実のみ抽出し表現は自作」を徹底すれば問題なし。**要修正点**: `summary` 作成者向けに「元サイトの文章を要約ツールでリライトする程度ではなく、事実を箇条書きに分解してから自分の文章で書き直す」手順を明文化すること |
| ④LLM生成文章の権利関係 | 文化庁の一般的見解では、人間の創作的関与が乏しい純粋なAI生成物は「著作物」に該当せず著作権が発生しない可能性がある。実務上の影響は「投稿文をそのまま転載されても著作権法上は差し止めにくい」程度で致命的ではないが、`terms.html` 第7条の「投稿コンテンツの著作権はユーザーに帰属」がAI投稿には当てはまらないため、4-7の条文案で手当てすること |

「summaryは人間が事前作成し、Claudeはその範囲内で言い換えるのみ」という制約は、事実面のハルシネーション対策として有効であり、著作権面でも意図しない大量引用や第三者コンテンツの丸ごと転記を防ぐ効果があり妥当。

### 4-6. 投稿の正確性リスク（ハルシネーション）— 要修正

現行設計は「summaryの範囲内のみ使用」「JSON形式検証」「temperature 0.5」を備えるが、**事後の訂正・削除フロー**が未整備。

1. **ユーザー通報導線**: 投稿詳細画面またはバッジのツールチップに「内容に誤りがあれば、お問い合わせよりご連絡ください」を明記し、`privacy.html` 第10条の問い合わせ先に直結
2. **内部モニタリング**: `ai_curator_logs` に加え、直近投稿を管理者が1日1回スポットチェック。運用手順書の時間割に1項目追加（例 07:35〜07:40）
3. **訂正の方法**: 単純削除ではなく「訂正のお知らせ」を付記する運用を原則とし（本文冒頭に「※本投稿には一部誤りがあり、訂正しました」）、金額・条文番号など実害の大きい誤りのみ削除対応
4. **責任の所在**: `terms.html` 第6条は「故意または重大な過失による損害は責任を負う／消費者契約法8条に基づき損害賠償を月額利用料相当に上限設定」という構成。**AI投稿は無料コンテンツであるためこの上限規定の実効性が乏しい**（月額支払額ゼロ＝上限もゼロと解釈されうる）。次項の条文案で「正確性を保証しない」旨を明記し、通常の過失に基づく責任をそもそも発生させない設計にすること

### 4-7. 利用規約への追記（条文案）

`terms.html` 第5条（EMUERおよびNFT）と第6条（免責事項）の間に新条文として挿入（挿入により第6条以降は繰り下げ番号となる点はエンジニアと調整）。
